# Development Workflow

How to work on ASU Global Connect day-to-day.

---

## Starting a Session

```bash
# 1. Open terminal and set environment
export PATH="$PATH:/Users/ashishrajshekhar/Library/Python/3.10/bin"
export TUTOR_ROOT="/Users/ashishrajshekhar/Desktop/2026/edx"

# 2. Start Docker Desktop (if not running)

# 3. Start services
tutor dev start

# 4. Wait for LMS to be ready (first request is slow)
tutor dev logs --follow lms
# Wait for: "Watching for file changes with StatReloader"
```

## Working on an XBlock

### Iteration loop

```
Edit code → Install → Test in browser → Check logs → Repeat
```

### 1. Edit code

Open `xblocks/<name>-xblock/` in your editor. XBlock files:

```
my_xblock/
├── my_xblock.py          ← Main XBlock class (Python)
├── static/
│   ├── html/student.html ← Student-facing template
│   ├── js/student.js     ← Frontend JavaScript
│   └── css/style.css     ← Styles
└── tests/
    └── test_xblock.py    ← Pytest tests
```

### 2. Install into containers

Since dev containers mount your local filesystem, a re-install is needed when you add new entry points (first time or after changing `setup.py`). For code-only changes, a restart is enough:

```bash
# First install (or after setup.py changes)
tutor dev run lms pip install -e /openedx/xblocks/my-xblock
tutor dev run cms pip install -e /openedx/xblocks/my-xblock
tutor dev restart lms cms

# For Python-only changes (no new entry points)
tutor dev restart lms cms

# For JS/CSS/HTML-only changes
# No restart needed — static files are served directly from the mounted path
```

### 3. Enable in Studio

One-time setup per course:
- Studio → Course → Advanced Settings → Advanced Module List
- Add: `"my_xblock"` (the entry point name from setup.py)
- Save

### 4. Add to a Unit

- Studio → Course Outline → Unit → Add Component → Advanced
- Select your XBlock
- It should appear in the unit

### 5. Preview in LMS

- Studio → View Live / Preview
- Or: LMS → your course → the unit

### 6. Check logs during testing

```bash
tutor dev logs --follow lms
# Watch for Python errors, 500s, handler calls
```

## Running Tests

```bash
cd $TUTOR_ROOT/xblocks/my-xblock

# Run all tests
python -m pytest tests/ -v

# With coverage report
python -m pytest tests/ --cov=my_xblock --cov-report=html

# Run a specific test file
python -m pytest tests/test_xblock.py -v

# Run a specific test
python -m pytest tests/test_xblock.py::TestMyXBlock::test_student_view -v
```

Coverage target: **80% minimum** per XBlock.

## Making Changes to LMS/CMS Settings

Settings live in `env/apps/openedx/settings/lms/development.py`. **Do not edit these directly** — they are regenerated by Tutor.

The correct way to add settings:
1. Create a Tutor plugin (for persistent changes)
2. Or temporarily edit and restart (not committed)

For simple dev hacks, you can edit `env/apps/openedx/settings/lms/development.py` directly, then:
```bash
tutor dev restart lms
```

But regenerating the env (`tutor config save`) will overwrite your changes.

## Checking What's Happening in Production-Like Mode

```bash
# Check Django management commands
tutor dev run lms ./manage.py lms help

# Check what's installed
tutor dev run lms pip list | grep -i xblock

# Check XBlock entry points
tutor dev run lms python -c "
import pkg_resources
for ep in pkg_resources.iter_entry_points('xblock.v1'):
    print(ep)
"
```

## Stopping a Session

```bash
tutor dev stop
```

Always stop cleanly to avoid MySQL redo log corruption.

> **Important:** If you need to force-quit Docker, do `tutor dev stop` first to let MySQL flush cleanly.
