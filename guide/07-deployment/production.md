# Production Deployment

Guide for deploying ASU Global Connect to production for the July 2026 pilot.

---

## Target: 500 Students, July 2026

The production environment must handle:
- 500 concurrent users (peak during class hours)
- All 6 XBlocks active
- ASU CAS SSO integration
- Canvas LTI grade passback
- Full HTTPS
- Data backup and recovery

---

## Pre-deployment Checklist

### Security
- [ ] `ENABLE_HTTPS: true` in config.yml
- [ ] SSL certificate provisioned (Caddy auto-TLS or bring your own)
- [ ] `DEBUG = False` in production settings (automatic in `tutor local`)
- [ ] `SECRET_KEY` is random and not committed to git
- [ ] `MYSQL_ROOT_PASSWORD` is strong and not the dev default
- [ ] `config.yml` not in any public repository
- [ ] All XBlock API keys (Mapbox, etc.) are production keys

### Infrastructure
- [ ] Server: 8-core CPU, 16 GB RAM, 100 GB SSD (minimum for 500 students)
- [ ] Docker and Docker Compose installed
- [ ] Tutor v21 installed
- [ ] DNS: LMS_HOST and CMS_HOST point to server IP

### Data
- [ ] MySQL backup strategy set up (daily dump to S3)
- [ ] MongoDB backup strategy set up
- [ ] Media storage on S3 (not local filesystem)

## Production Config Changes

Update `config.yml` for production:

```bash
tutor config save \
  --set LMS_HOST="openedx.asu.edu" \
  --set CMS_HOST="studio.openedx.asu.edu" \
  --set CONTACT_EMAIL="ashish@asu.edu" \
  --set ENABLE_HTTPS=true

tutor config save  # regenerate env/
```

---

## Deploying

```bash
# First-time production launch
tutor local launch

# Subsequent starts
tutor local start

# Run migrations after code changes
tutor local run lms ./manage.py lms migrate
tutor local run cms ./manage.py cms migrate

# Install XBlocks in production
tutor local run lms pip install git+https://github.com/asu-coral/gamification-xblock.git
tutor local run cms pip install git+https://github.com/asu-coral/gamification-xblock.git
tutor local restart lms cms

# Create admin user
tutor local do createuser --staff --superuser admin admin@openedx.local
```

## S3 Media Storage

In production, uploaded files go to S3, not the local filesystem:

```bash
tutor config save \
  --set OPENEDX_AWS_ACCESS_KEY="AKIAIOSFODNN7EXAMPLE" \
  --set OPENEDX_AWS_SECRET_ACCESS_KEY="..." \
  --set S3_STORAGE_BUCKET="asu-global-connect-media"
```

Or use MinIO (self-hosted S3-compatible):
```bash
tutor plugins install minio
tutor plugins enable minio
tutor config save
```

INSTALLED_APPS += ['django_cas_ng']

    'django_cas_ng.backends.CASBackend',

## Monitoring

```bash
# Check all containers
tutor local dc ps

# Follow LMS logs
tutor local logs --follow lms

# Check error rate
tutor local logs --follow lms 2>&1 | grep -E "ERROR|500"
```

For production, set up:
- Sentry (error tracking): add `sentry-sdk` to INSTALLED_APPS
- Prometheus + Grafana (metrics): Tutor has a monitoring plugin
- Uptime monitoring: set up a heartbeat check at `/heartbeat`

## Scaling for 500 Students

At 500 concurrent users, the single-container setup should hold, but monitor:
- LMS memory usage (Django + uWSGI workers) — target < 8 GB
- MySQL connection pool — uWSGI has `max-requests` to recycle workers
- Redis memory — sessions + leaderboard should fit in < 1 GB
- MongoDB — course structure reads are cached; should be fast

If more capacity is needed:
- Increase uWSGI workers: `tutor config save --set UWSGI_WORKERS=8`
- Add a second LMS container (read replicas, load-balanced by Caddy)
- Consider managed MySQL (RDS) for better I/O performance
